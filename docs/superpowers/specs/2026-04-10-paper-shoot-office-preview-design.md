# PaperShoot Office Preview and Art Alignment Design

**Date:** 2026-04-10  
**Project:** PaperShoot  
**Goal:** 모든 플레이 스테이지를 공용 오피스 공간 안에서 일관되게 보이도록 재구성하고, `?stage=` 직접 진입과 Playwright 검수 경로를 통해 각 스테이지의 실제 모습을 즉시 확인 가능하게 만든다.

## 1. 문제 정의

현재 구현은 다음 세 가지 문제를 갖고 있다.

1. Stage 1이 `room`으로 정의되어 있고 씬이 항상 `stageCatalog[0]`만 부팅하기 때문에, office 테마를 넣더라도 실제 플레이에서는 보이지 않는다.
2. 배경과 장애물의 시각 표현이 임시 도형과 placeholder 리소스 위주라서 참조 게임 수준의 오피스 공간감이 읽히지 않는다.
3. 특정 스테이지를 확인하려면 실제 진행을 통해 도달해야 하므로, 개발과 유지보수 관점에서 각 스테이지의 모습을 빠르게 검수할 수 없다.

이번 디자인의 목적은 `플레이어용 스테이지 선택 UI 추가`가 아니라, `공용 오피스 기반의 실제 플레이 화면을 유지하면서 개발/검수용 직접 진입과 시각 리소스 정합성을 확보하는 것`이다.

## 2. 고정 결정

이번 작업에서 아래 결정은 확정된 것으로 본다.

- Stage 1~6 전체를 모두 `office` 세계 안에서 변주한다.
- 배경은 스테이지마다 별도 공간으로 나누지 않고, `공용 오피스 배경 세트 1개`를 유지한다.
- 스테이지 차이는 `바람`, `장애물 종류`, `장애물 배치`, `난이도 밀도`로 만든다.
- 플레이어가 임의로 스테이지를 고르는 UI는 만들지 않는다.
- 개발/QA/Playwright 전용으로만 `?stage=1` ~ `?stage=6` 직접 진입을 지원한다.
- 배경과 장애물은 SVG 임시 도형이 아니라 래스터 이미지 리소스로 교체한다.
- 이미지 리소스는 `시뮬레이션 기하를 따라야 하며`, 특히 쓰레기통과 장애물은 실제 충돌체와 시각적 외곽이 어긋나면 안 된다.

## 3. 제품 방향

### 3.1 공간 정체성

PaperShoot는 더 이상 `room`, `classroom`, `cafe`처럼 서로 다른 공간을 점프하는 게임이 아니다.  
모든 스테이지는 같은 오피스 공간 안에서 벌어지는 종이공 던지기 챌린지다.

이 오피스는 `스타일라이즈된 3D 게임풍`으로 표현한다.  
즉, 실사 복제를 목표로 하지 않지만 아래 정보는 첫눈에 읽혀야 한다.

- 큐비클과 사무실 벽면
- 바닥과 통로
- 책상과 모니터, 서랍장, 의자 같은 사무용 가구
- 화면 아래 시작 지점과 화면 안쪽 목표 지점의 거리감

### 3.2 공용 배경 전략

공간은 하나의 공용 오피스 세트로 유지한다.

- 원경: 큐비클, 바닥, 뒤쪽 책상, 조명
- 중경: 책상/모니터/가구 실루엣
- 전경: 시작 지점과 깊이감을 만드는 책상 모서리

이 공용 세트 위에 쓰레기통, 선풍기, 장애물만 바꿔서 Stage 1~6을 만든다.  
따라서 난이도와 시각 차이는 `공간 전환`이 아니라 `플레이 레인 변화`에서 읽혀야 한다.

## 4. 직접 진입과 검수 흐름

### 4.1 직접 진입 규칙

게임은 URL 파라미터 `?stage=N`을 읽어 스테이지 오버라이드를 허용한다.

- 허용 범위: `1` ~ `6`
- `?stage=`가 없으면 기본 진행 경로를 사용한다.
- 잘못된 값이면 기본 진행 경로 또는 Stage 1로 안전하게 폴백한다.

이 기능은 개발/QA/Playwright 용도이며, 별도의 플레이어용 스테이지 선택 UI는 제공하지 않는다.

### 4.2 Playwright 검수 규칙

Playwright는 `?stage=1`부터 `?stage=6`까지 모든 스테이지를 직접 열어 실제 장면을 검수해야 한다.

각 스테이지 공통 검수 항목:

1. 해당 스테이지가 정상 부팅된다.
2. HUD에 현재 Stage 번호가 맞게 표시된다.
3. 공용 오피스 배경과 해당 스테이지 장애물이 함께 보인다.
4. 최소 1회 드래그-릴리즈 투척이 동작한다.
5. 대표 스크린샷이 저장된다.

공통 회귀 항목:

- Stage 1 또는 지정된 대표 스테이지에서 `Game Over -> Retry` 흐름이 동작한다.
- Stage 6에서 최종 장애물 조합과 바람 표시가 함께 보인다.

즉, 각 스테이지는 더 이상 “직접 플레이로 도달했을 때만 눈으로 확인하는 상태”가 아니어야 한다.

## 5. 이미지 생성 전략

### 5.1 원칙

이미지는 `충돌체를 바꾸는 기준`이 아니라 `충돌체를 표현하는 표면`이다.

따라서 생성 순서는 다음과 같다.

1. 시뮬레이션 기하를 소스 오브 트루스로 고정한다.
2. 카메라 투영 기준으로 대표 화면 크기와 앵커를 계산한다.
3. 이 기하 시트에 맞춰 이미지를 생성한다.
4. 렌더러는 월드 좌표와 앵커 기준으로 이미지를 배치한다.

이 순서를 지키지 않으면 쓰레기통 입구와 실제 성공 판정, 장애물 외곽과 실제 충돌 박스가 어긋난다.

### 5.2 스타일

이미지는 `스타일라이즈된 3D 게임풍`으로 생성한다.

- 오피스는 정리된 큐비클 환경으로 읽혀야 한다.
- 과한 실사 노이즈보다는 명확한 실루엣과 색 분리가 우선이다.
- 쓰레기통, 장애물, 선풍기는 모바일에서도 가독성이 유지되어야 한다.
- 광원과 원근 방향은 세트 전체에서 일관되어야 한다.

## 7. 치수표와 앵커 시트

### 7.1 시트에 반드시 들어갈 항목

모든 충돌 대상은 아래 필드를 갖는 기준 시트를 먼저 작성한다.

- `worldSize`
  - 월드 좌표 기준 폭/높이/깊이
- `anchor`
  - 이미지 배치 기준점
- `collisionShape`
  - 실제 판정 형상
- `referenceStage`
  - 대표 검수 스테이지
- `expectedScreenSize`
  - 대표 스테이지에서 예상되는 화면상 픽셀 크기
- `silhouetteRule`
  - 이미지 외곽이 충돌체를 얼마나 타이트하게 감쌀지에 대한 규칙

### 7.2 우선 정합 대상

가장 엄격하게 맞춰야 하는 대상은 아래 순서다.

1. `trash-bin-main`
2. `obstacle-center-block`
3. `obstacle-dual-block-left`
4. `obstacle-dual-block-right`
5. `obstacle-moving-cart`
6. `obstacle-swing-panel`
7. `obstacle-narrow-gate`

### 7.3 앵커 규칙

- 쓰레기통: `bottom-center`
- 고정 블록: `center`
- 이동 카트: `bottom-center`
- 스윙 패널: `top-center` 또는 힌지 기준 앵커
- 좁은 게이트: `center`

### 7.4 쓰레기통 정합성 규칙

쓰레기통은 시각적 과장이 허용되더라도 아래 항목은 실제 판정과 맞아야 한다.

- 입구 폭
- 입구 높이
- 림 위치
- 내부가 시작되는 상단 경계

플레이어는 보이는 입구를 보고 던지므로, 여기 어긋남은 허용하지 않는다.

### 7.5 장애물 정합성 규칙

장애물은 시각 실루엣이 실제 충돌 박스보다 과도하게 커 보이면 안 된다.

- 장식 돌출은 허용 가능
- 충돌체보다 좁아 보이는 것도 금지
- `narrow-gate`는 시각적으로 보이는 통과 폭과 실제 판정 폭이 일치해야 한다

## 8. 스테이지 매핑

Stage 1~6은 모두 같은 오피스 세트 위에서 동작한다.

- Stage 1: 장애물 없음, 약한 바람
- Stage 2: 장애물 없음, 강한 바람
- Stage 3: `obstacle-center-block`
- Stage 4: `obstacle-dual-block-left/right`
- Stage 5: `obstacle-moving-cart`
- Stage 6: `obstacle-swing-panel` + `obstacle-narrow-gate`

즉, 스테이지 차이는 `공간 종류`가 아니라 `같은 공간 안 레인 구성의 변화`다.

## 9. 기술 설계 영향

### 9.1 Stage 데이터

- 모든 Stage의 `theme`는 `office`로 통일한다.
- 현재 `room`, `classroom`, `cafe` 값은 제거하거나 더 이상 사용하지 않는 방향으로 정리한다.

### 9.2 Stage 진입 해석기

- URL에서 `stage` 파라미터를 읽어 현재 스테이지를 결정하는 해석기를 둔다.
- 이 해석기는 씬 생성 시점 전에 적용되어야 하며, Playwright가 동일한 경로를 사용할 수 있어야 한다.

### 9.3 렌더러

- 공용 오피스 배경 레이어를 먼저 렌더링한다.
- 그 위에 공용 플레이 오브젝트와 해당 스테이지 장애물을 올린다.
- 쓰레기통과 장애물은 기하 시트 기준 앵커로만 배치한다.

## 10. 비목표

이번 범위에 포함하지 않는 항목은 다음과 같다.

- 플레이어용 스테이지 선택 메뉴
- 스테이지마다 다른 배경 공간 세트 제작
- 실사풍 그래픽 복제
- 충돌체를 이미지에 맞춰 다시 설계하는 작업

## 11. 성공 기준

이번 디자인이 성공했다고 판단하는 기준은 아래와 같다.

1. Stage 1부터 실제로 오피스 공간이 읽힌다.
2. Stage 1~6을 `?stage=` 직접 진입으로 모두 확인할 수 있다.
3. Playwright가 Stage 1~6 전부를 직접 진입 기준으로 검수한다.
4. 쓰레기통과 장애물의 보이는 실루엣이 실제 충돌체와 어긋나지 않는다.
5. 스테이지 차이가 배경 교체가 아니라 장애물/바람/배치 차이로 읽힌다.
