# PaperShoot 설계 문서

- 작성일: 2026-04-09
- 문서 성격: 브레인스토밍 승인 결과를 고정한 구현 전 설계 문서
- 프로젝트 상태: 신규 웹 게임 프로젝트 기준 설계

## 1. 프로젝트 개요

`PaperShoot`는 뭉친 종이를 쓰레기통에 던지는 스테이지 클리어형 웹 게임이다. 플레이어는 정면의 쓰레기통을 향해 각도와 파워를 조절해 종이를 던지며, 선풍기가 만드는 바람과 각종 장애물을 읽어 제한된 투척 횟수 안에 목표 성공 횟수를 달성해야 한다.

핵심 재미는 다음 세 가지다.

1. 바람 방향과 세기를 읽고 던지는 계산형 재미
2. 제한 횟수 안에서 해답을 찾아가는 퍼즐 아케이드 감각
3. 스테이지마다 다른 공간 테마와 기믹 조합을 해석하는 학습형 진행

## 2. 최종 확정된 방향

본 문서는 아래 결정을 기준으로 한다.

- 게임 구조: 스테이지 클리어형
- 클리어 규칙: 제한 횟수 안에 목표 성공 횟수 달성
- 조작 방식: 각도 조절 + 파워 게이지형
- 카메라: 준 FPS형 정면 투척 구도
- 바람 정보: 완전 예측형
- 기믹 성격: 멀티 기믹형
- 물리 톤: 하이브리드형
- 판정 톤: 중간 판정
- 플랫폼: PC 웹 / 모바일 웹 동시 대응
- 공간 테마: 일상 혼합 테마
- 손 모델: 노출하지 않음

## 3. 목표와 비목표

### 3.1 목표

- 플레이어가 짧은 시간 안에 규칙을 이해하고 재도전할 수 있어야 한다.
- 바람과 장애물을 읽고 해법을 찾는 퍼즐성이 분명해야 한다.
- 모바일과 PC에서 모두 동일한 규칙과 난이도를 유지해야 한다.
- 실패 원인이 즉시 읽혀야 하며, 억울한 실패보다 계산 오차로 느껴져야 한다.
- 첫 버전은 적은 수의 기믹으로도 확장 가능한 구조를 갖춰야 한다.

### 3.2 비목표

- 실제 3D 물리 기반의 완전한 FPS 투척 게임을 목표로 하지 않는다.
- 손 애니메이션, 무기형 UI, 현실형 1인칭 몰입 연출을 우선하지 않는다.
- 업그레이드, 재화, 상점, 메타 성장 시스템은 첫 버전 범위에 포함하지 않는다.
- 리플레이, 리더보드, 실시간 경쟁 요소는 MVP 범위에서 제외한다.

## 4. 플레이어 경험 목표

플레이어는 스테이지에 진입하면 먼저 정면의 쓰레기통, 선풍기 위치, 장애물 배치, HUD에 표시된 바람 정보를 읽는다. 이후 각도를 맞추고 파워 게이지 타이밍을 맞춰 종이를 던진다. 투척 후에는 종이가 바람 구역과 충돌 오브젝트의 영향을 받아 궤적이 변하고, 플레이어는 결과를 보고 바로 보정 시도를 한다.

각 스테이지는 짧은 시간 안에 실패 원인을 파악하고 다시 도전할 수 있어야 한다. 즉, 이 게임의 감정 곡선은 긴 몰입형 체험보다 `읽기 -> 시도 -> 실패 원인 이해 -> 재조정 -> 성공`의 반복에 가깝다.

## 5. 코어 게임 루프

1. 스테이지 진입
2. 목표 조건과 남은 투척 수 확인
3. 바람 방향/세기와 장애물 구조 판독
4. 각도 조절
5. 파워 게이지 확정
6. 종이 투척 시뮬레이션 진행
7. 성공 또는 실패 판정
8. 남은 투척 수가 있으면 즉시 재시도, 없으면 실패 처리
9. 목표 달성 시 다음 스테이지 해금 또는 이동

이 루프는 한 번의 시도가 짧고 명확해야 한다. 한 투척에서 너무 많은 정보가 동시에 바뀌면 공정성이 떨어지므로, 스테이지별 신규 규칙은 1개씩만 추가한다.

## 6. 카메라와 화면 구도

### 6.1 카메라 원칙

카메라는 `준 FPS형 정면 투척`으로 고정한다.

- 화면 중심은 정면의 쓰레기통이다.
- 플레이어는 정면으로 던지는 느낌을 받아야 한다.
- 손 모델은 노출하지 않는다.
- 종이 출발점은 화면 하단 중앙 근처의 제한된 시각 요소로만 암시한다.
- 플레이어가 읽어야 할 주요 정보는 정면 깊이 공간 위에 배치된다.

### 6.2 시점 설계 이유

이 시점은 순수 1인칭보다 시야를 넓게 확보할 수 있고, 종이와 손의 연출 비용을 줄이며, 퍼즐 아케이드 구조에 더 적합하다. 또한 2D/2.5D 방식의 단순화된 시뮬레이션과도 잘 맞는다.

## 7. 조작 설계

### 7.1 입력 모델

- 각도 조절: 마우스 드래그, 키 입력, 또는 터치 드래그 기반
- 파워 확정: 타이밍 입력 또는 버튼 탭
- 재시도: 즉시 동일 스테이지 재도전 가능

### 7.2 조작 흐름

1. 플레이어가 각도를 맞춘다.
2. 파워 게이지가 왕복 또는 상승 동작을 한다.
3. 플레이어가 원하는 시점에 입력해 파워를 확정한다.
4. 종이가 투척된다.

### 7.3 조작 목표

- PC와 모바일에서 같은 룰을 유지하되, 입력 수단만 다르게 처리한다.
- 조작은 어렵지 않지만 숙련 차이는 존재해야 한다.
- 고의적 난해함이 아니라 계산 가능한 오차 범위를 제공해야 한다.

## 8. HUD와 정보 가독성

### 8.1 HUD 배치 권장안

현 시점 기준 권장 HUD는 다음과 같다.

- 상단: 스테이지 번호, 목표 조건, 남은 투척 수
- 우측 상단: 바람 HUD
- 하단: 각도 표시 + 파워 게이지

### 8.2 바람 HUD

바람 HUD에는 아래 정보를 항상 표시한다.

- 방향 화살표
- 세기 수치
- 강도 라벨 (`약함`, `보통`, `강함` 등)

HUD는 전역 정보를 제공하고, 플레이필드 내부는 공간 정보를 제공한다.

- HUD 역할: 현재 바람 상태를 빠르게 확인
- 플레이필드 역할: 어느 구역에서 어떤 방식으로 바람이 작동하는지 시각적으로 이해

### 8.3 플레이필드 시각 정보

- 선풍기 위치
- 바람 영향 구역 파티클 또는 스트릭 효과
- 장애물의 현재 상태
- 쓰레기통 위치와 입구 가시성
- 필요 시 초반 스테이지만 제한된 보조선 제공

### 8.4 정보 표현 원칙

- 실패 원인은 반드시 화면에서 읽혀야 한다.
- 텍스트보다 공간 배치와 간결한 아이콘을 우선한다.
- 모바일에서는 HUD가 시야를 가리지 않아야 한다.

## 9. 물리와 판정 톤

### 9.1 물리 톤

물리 톤은 `하이브리드형`으로 설계한다.

- 기본은 읽기 쉬운 아케이드 성격
- 후반으로 갈수록 바람과 장애물의 영향이 더 크게 느껴짐
- 현실처럼 보이되, 밸런싱 가능한 단순 모델을 유지

### 9.2 판정 톤

판정은 `중간 판정`으로 한다.

- 각도와 파워를 맞춰야 하지만 약간의 오차는 허용
- 통 입구 근처 성공은 과도하게 빡빡하지 않게 처리
- 완전 자동 흡입처럼 보이지는 않도록 조심

### 9.3 실패 해석 가능성

플레이어는 다음 중 무엇 때문에 실패했는지 즉시 알 수 있어야 한다.

- 바람에 밀렸다.
- 파워가 과했다.
- 파워가 부족했다.
- 장애물에 막혔다.
- 입구 각도가 맞지 않았다.

## 10. 기믹 구조와 난이도 곡선

### 10.1 기믹 분류

#### 바람 기믹
- 고정 선풍기
- 회전 선풍기
- 특정 구역에서만 적용되는 바람 구역

#### 경로 방해 기믹
- 고정 장애물
- 이동 장애물
- 좁은 입구 또는 부분 가림 구조

#### 판독 보조 기믹
- 바람 파티클
- 영향 구역 시각화
- 초반 전용 투척 보조 요소

### 10.2 난이도 상승 원칙

스테이지 난이도는 아래 흐름으로 상승한다.

1. 약한 단일 바람
2. 강한 단일 바람
3. 강한 바람 + 고정 장애물
4. 복수 바람 구역 또는 경로 판독 강조
5. 바람 + 이동 장애물
6. 강한 바람 + 이동 장애물 + 좁은 입구 조합

### 10.3 설계 규칙

- 한 스테이지에서 새로 배우는 규칙은 1개만 추가한다.
- 기존 기믹을 다음 스테이지에서 조합해 난도를 올린다.
- 플레이어가 읽지 못해 실패하는 구간을 만들지 않는다.
- 멀티 기믹형이더라도, 문제를 푸는 과정은 단계적으로 학습되게 한다.

## 11. 공간 테마 전략

공간 테마는 일상 혼합형으로 설계한다.

예시 테마는 다음과 같다.

- 방
- 사무실
- 교실
- 카페

중요한 점은 아트 테마가 달라도 기믹 역할은 재사용 가능해야 한다는 것이다. 예를 들어 사무실의 책 더미와 카페의 컵 진열대는 외형은 다르지만 `고정 장애물`이라는 기능은 동일하게 유지할 수 있다.

## 12. 시스템 아키텍처

### 12.1 권장 기술 스택

- Phaser
- TypeScript
- Vite
- DOM 오버레이 기반 HUD

### 12.2 스택 선택 이유

이 게임은 자유로운 3D 이동이 핵심이 아니라, 정면 투척 느낌을 가진 퍼즐 아케이드 구조가 핵심이다. 따라서 진짜 3D 엔진보다, 빠르고 안정적인 2D/2.5D 연출과 입력/시뮬레이션 제어가 유리하다. Phaser는 이 조건에 맞고, 웹 배포와 입력 대응도 단순하다.

### 12.3 모듈 구조

#### GameShell
- 게임 부팅
- 화면 리사이즈 대응
- 씬 전환 및 앱 수준 상태 관리

#### StageRuntime
- 현재 스테이지 규칙 유지
- 남은 투척 수 관리
- 성공/실패 판정
- 스테이지 종료 처리

#### AimInput
- 마우스, 터치, 키 입력을 각도/파워 입력으로 변환
- 플랫폼별 입력 차이를 통합

#### ThrowSimulation
- 종이의 비행 계산
- 포물선, 횡방향 편차, 낙하 감쇠 처리
- 쓰레기통 도달 판정

#### WindSystem
- 선풍기와 바람 구역의 방향/세기 계산
- 현재 HUD에 보여줄 바람 값 제공
- 구역 진입 시 종이 궤적에 영향 적용

#### ObstacleSystem
- 고정 장애물
- 이동 장애물
- 좁은 입구
- 기타 충돌체 관리

#### HudPresenter
- 상단 목표/진행 정보 표시
- 우측 바람 HUD 표시
- 하단 각도/파워 UI 표시

#### StageRepository
- 스테이지 데이터를 로드하고 제공
- 스테이지별 규칙과 배치를 코드와 분리

### 12.4 아키텍처 원칙

- 시뮬레이션과 렌더링을 분리한다.
- 저장 가능한 상태는 렌더러 객체가 아니라 시뮬레이션 데이터여야 한다.
- HUD는 DOM 레이어로 유지해 텍스트 가독성을 확보한다.
- 스테이지 규칙은 데이터 중심으로 관리한다.

## 13. 시뮬레이션 모델

이 게임은 진짜 3D 탄도 시뮬레이션보다 `정면 투척 느낌을 주는 단순화 모델`이 중요하다.

내부 시뮬레이션은 최소한 아래 축을 유지하면 된다.

- 전진 거리
- 좌우 편차
- 높이

바람은 특정 구역에 진입했을 때 좌우 편차와 낙하 타이밍에 영향을 주는 방식으로 처리한다. 이 구조는 화면상으로는 정면 투척처럼 보이면서도, 밸런싱과 테스트를 안정적으로 만들 수 있다.

## 14. 스테이지 데이터 구조

스테이지는 코드 하드코딩이 아니라 데이터로 정의한다. 최소 필드는 다음과 같다.

- `stageId`
- `theme`
- `throwLimit`
- `successTarget`
- `binPosition`
- `windZones`
- `obstacles`
- `cameraPreset`
- `assistOptions`
- `clearConditionText`

예상되는 의사 구조는 아래와 같다.

```ts
interface StageConfig {
  stageId: string;
  theme: 'room' | 'office' | 'classroom' | 'cafe';
  throwLimit: number;
  successTarget: number;
  binPosition: {
    x: number;
    y: number;
    depth: number;
  };
  windZones: WindZoneConfig[];
  obstacles: ObstacleConfig[];
  cameraPreset: 'semi-fps-default';
  assistOptions: {
    showGuideArc: boolean;
    showImpactHint: boolean;
  };
  clearConditionText: string;
}
```

## 15. 에러 처리와 피드백

### 15.1 플레이 피드백

- 성공 시 즉시 시각/청각 피드백 제공
- 실패 시 원인을 읽을 수 있는 연출 제공
- 남은 투척 수가 줄어드는 것을 명확히 표시
- 목표 달성 시 다음 스테이지 진입을 자연스럽게 연결

### 15.2 시스템 레벨 대응

- 스테이지 데이터 오류 시 안전한 실패 처리 필요
- 잘못된 장애물 배치나 바람 구역 값은 로드 단계에서 검증 필요
- 모바일 해상도에서 HUD 겹침 발생 시 레이아웃 축소 규칙 필요

## 16. 테스트 전략

### 16.1 검증 목표

- 같은 입력이면 항상 같은 결과가 나오는 결정적 시뮬레이션 유지
- 클리어 가능한 스테이지 설계 보장
- 모바일/PC에서 동일 규칙 유지
- HUD 가독성 유지

### 16.2 핵심 테스트 항목

- 바람 구역 진입 시 의도한 편차가 적용되는가
- 쓰레기통 성공/실패 판정이 일관적인가
- 이동 장애물 주기와 클리어 조건이 충돌하지 않는가
- 각도/파워 입력이 플랫폼별로 같은 논리로 동작하는가
- 모바일에서 HUD와 플레이필드가 겹치지 않는가

## 17. MVP 범위

### 17.1 포함 범위

- 스테이지 순차 진행 또는 기본 선택 화면
- 준 FPS형 정면 투척 화면
- 각도 조절 + 파워 게이지 투척
- 우측 상단 바람 HUD
- 선풍기 기반 바람 구역
- 제한 투척 수 / 성공 목표
- 성공/실패 결과 화면
- PC/모바일 반응형 대응
- 최소 5~8개 스테이지

### 17.2 제외 범위

- 손 애니메이션
- 리플레이 시스템
- 메타 성장 시스템
- 과도한 반사 물리
- 실시간 랜덤 풍향 변화
- 상점, 업그레이드, 재화 시스템
- 온라인 요소

## 18. 초기 스테이지 구성

### Stage 1
- 단일 선풍기
- 약한 바람
- 장애물 없음
- 목표: 3번 안에 1회 성공
- 학습 목표: 바람 HUD와 기본 투척 감각 이해

### Stage 2
- 단일 선풍기
- 더 강한 바람
- 장애물 없음
- 학습 목표: 바람 세기에 따른 보정 이해

### Stage 3
- 단일 선풍기
- 강한 바람
- 고정 장애물 1개
- 학습 목표: 직선 경로가 항상 정답이 아님을 이해

### Stage 4
- 복수 바람 구역 또는 확장된 바람 영향 범위
- 고정 장애물 1~2개
- 학습 목표: 경로 해석 시작

### Stage 5
- 단일 바람 + 이동 장애물
- 학습 목표: 타이밍 요소 도입

### Stage 6~8
- 강한 바람 + 이동 장애물 + 좁은 입구 조합
- 학습 목표: 멀티 기믹 조합 퍼즐 완성

## 19. 출시 판단 기준

아래 기준이 충족되면 첫 버전의 방향성이 맞는 것으로 본다.

- Stage 1~3에서 규칙이 직관적으로 학습된다.
- 실패 원인이 플레이어에게 즉시 읽힌다.
- 모바일과 PC에서 입력 불공정이 크지 않다.
- 중반 이후 스테이지에서 계산형 퍼즐 재미가 살아난다.
- 바람 HUD가 과잉 설명 없이도 충분히 유용하다.

## 20. 후속 계획 방향

권장 진행 순서는 아래와 같다.

1. Stage 1~3 수준의 핵심 루프만 먼저 구현
2. 바람 수치와 판정 감각 튜닝
3. Stage 4~5로 기믹 확장
4. Stage 6~8로 멀티 기믹 조합 완성
5. 테마 아트 다양화

## 21. 결론

`PaperShoot`의 첫 구현은 `Phaser 기반의 준 FPS형 정면 투척 퍼즐 아케이드`로 정의한다. 핵심은 진짜 3D가 아니라, 읽기 쉬운 바람 정보와 계산 가능한 투척 결과를 중심으로 한 퍼즐 구조다. 첫 버전은 작은 범위의 바람/장애물 조합으로 완성도를 확보하고, 이후 테마와 기믹을 확장하는 방식이 가장 현실적이다.
