# PaperShoot 설계 문서

- 작성일: 2026-04-09
- 최종 수정일: 2026-04-10
- 문서 성격: 브레인스토밍 승인 결과를 고정한 구현 전 설계 문서
- 프로젝트 상태: 신규 웹 게임 프로젝트 기준 설계

## 1. 프로젝트 개요

`PaperShoot`는 뭉친 종이를 쓰레기통에 던지는 스테이지 클리어형 웹 게임이다. 플레이어는 정면의 쓰레기통을 향해 각도와 파워를 조절해 종이를 던지며, 선풍기가 만드는 바람과 각종 장애물을 읽어 제한된 투척 횟수 안에 목표 성공 횟수를 달성해야 한다.

핵심 재미는 다음 세 가지다.

1. 바람 방향과 세기를 읽고 던지는 계산형 재미
2. 제한 횟수 안에서 해답을 찾아가는 퍼즐 아케이드 감각
3. 공간 배치와 장애물 구조를 해석하며 점진적으로 학습하는 진행

## 2. 최종 확정된 방향

본 문서는 아래 결정을 기준으로 한다.

- 게임 구조: 스테이지 클리어형
- 클리어 규칙: 제한 횟수 안에 목표 성공 횟수 달성
- 점수 규칙: `들어갔는가 / 안 들어갔는가`만 판단하는 이진 판정
- 진행 구조: 순차 진행형
- 조작 방식: 각도 조절 + 파워 게이지형
- 입력 플로우: 조준 단계와 파워 단계를 분리한 2단계 확정형 입력
- 카메라: 준 FPS형 정면 투척 구도
- 바람 정보: 완전 예측형
- 바람 규칙: 스테이지당 단일 선풍기 1개만 사용
- 기믹 성격: 단일 선풍기 바람 + 장애물 조합형
- 물리 톤: 하이브리드형 아케이드
- 판정 톤: 중간 판정
- 실패 규칙: 바닥에 닿는 순간 즉시 실패
- 재시도 규칙: 종이와 입력 상태만 초기화, 월드 시간과 장애물 상태는 유지
- 플랫폼: PC 웹 / 모바일 웹 동시 대응
- 공간 테마: 일상 혼합 테마
- 손 모델: 노출하지 않음

## 3. 목표와 비목표

### 3.1 목표

- 플레이어가 짧은 시간 안에 규칙을 이해하고 재도전할 수 있어야 한다.
- 바람과 장애물을 읽고 해법을 찾는 퍼즐성이 분명해야 한다.
- 모바일과 PC에서 모두 동일한 규칙과 난이도를 유지해야 한다.
- 실패 원인이 즉시 읽혀야 하며, 억울한 실패보다 계산 오차로 느껴져야 한다.
- 첫 버전은 적은 수의 기믹으로도 확장 가능한 구조를 갖춰야 한다.
- 종이의 비행과 충돌이 물리적으로 완벽할 필요는 없지만, 플레이어가 보기에는 자연스럽고 일관되어야 한다.

### 3.2 비목표

- 실제 3D 물리 기반의 완전한 FPS 투척 게임을 목표로 하지 않는다.
- 손 애니메이션, 현실형 1인칭 몰입 연출을 우선하지 않는다.
- 업그레이드, 재화, 상점, 메타 성장 시스템은 첫 버전 범위에 포함하지 않는다.
- 리플레이, 리더보드, 실시간 경쟁 요소는 MVP 범위에서 제외한다.
- 복수 선풍기, 복수 풍향, 랜덤 풍향 전환은 첫 버전에서 다루지 않는다.
- 점수 보너스, 별점, 스타일 점수, 콤보 시스템은 첫 버전에서 다루지 않는다.

## 4. 플레이어 경험 목표

플레이어는 스테이지에 진입하면 먼저 정면의 쓰레기통, 선풍기 위치, 장애물 배치, HUD에 표시된 바람 정보를 읽는다. 이후 각도를 맞추고 파워 게이지 타이밍을 맞춰 종이를 던진다. 투척 후에는 종이가 바람 영향 범위와 충돌 오브젝트의 영향을 받아 궤적이 변하고, 플레이어는 결과를 보고 바로 보정 시도를 한다.

각 스테이지는 짧은 시간 안에 실패 원인을 파악하고 다시 도전할 수 있어야 한다. 즉, 이 게임의 감정 곡선은 긴 몰입형 체험보다 `읽기 -> 시도 -> 실패 원인 이해 -> 재조정 -> 성공`의 반복에 가깝다.

## 5. 코어 게임 루프

1. 스테이지 진입
2. 목표 조건과 남은 투척 수 확인
3. 단일 선풍기의 방향/세기와 장애물 구조 판독
4. 조준 단계에서 각도 조절
5. 파워 단계에서 게이지 확정
6. 종이 투척 시뮬레이션 진행
7. 성공 또는 실패 판정
8. 남은 투척 수가 있으면 즉시 재시도, 없으면 실패 처리
9. 목표 달성 시 다음 스테이지로 이동

이 루프는 한 번의 시도가 짧고 명확해야 한다. 한 투척에서 너무 많은 정보가 동시에 바뀌면 공정성이 떨어지므로, 스테이지별 신규 규칙은 1개씩만 추가한다.

`성공/실패 결과 화면`은 월드를 멈추는 별도 씬이 아니라 짧은 오버레이 피드백으로 처리한다. 실패 시에는 약 `300~500 ms` 동안 실패 원인만 보여준 뒤 즉시 같은 월드 상태에서 다시 조준 가능해야 하며, 성공 시에는 약 `600~900 ms` 동안 성공 피드백을 보여주고 다음 스테이지로 연결한다.

## 6. 카메라와 화면 구도

### 6.1 카메라 원칙

카메라는 `준 FPS형 정면 투척`으로 고정한다.

- 화면 중심은 정면의 쓰레기통이다.
- 플레이어는 정면으로 던지는 느낌을 받아야 한다.
- 손 모델은 노출하지 않는다.
- 종이 출발점은 화면 하단 중앙 근처의 제한된 시각 요소로만 암시한다.
- 플레이어가 읽어야 할 주요 정보는 정면 깊이 공간 위에 배치된다.

### 6.2 시점 설계 이유

이 시점은 순수 1인칭보다 시야를 넓게 확보할 수 있고, 손 연출 비용을 줄이며, 퍼즐 아케이드 구조에 더 적합하다. 또한 2.5D 방식의 단순화된 시뮬레이션과도 잘 맞는다.

## 7. 조작 설계

### 7.1 입력 모델

- 조준 단계와 파워 단계를 분리한 `2단계 확정형 입력`을 사용한다.
- 각도 조절: PC는 마우스 이동 또는 드래그, 모바일은 화면 드래그
- 파워 시작/확정: PC는 `Space` 또는 클릭, 모바일은 전용 버튼 탭
- 재시도: 즉시 동일 스테이지 재도전 가능
- 조준 입력은 화면 전체가 아니라 제한된 조준 영역 안에서만 유효하다.

### 7.2 조작 흐름

1. 플레이어가 조준점을 움직여 각도를 맞춘다.
2. 플레이어가 `던지기` 입력을 눌러 파워 게이지를 시작한다.
3. 파워 게이지는 `왕복형(ping-pong)`으로 움직인다.
4. 플레이어가 다시 입력해 현재 파워를 확정한다.
5. 종이가 즉시 투척된다.

중요한 구현 원칙:
- 조준 단계와 파워 단계에서 움직이는 것은 `실제 물리 종이`가 아니라 조준 프리뷰다.
- 실제 충돌과 물리 계산을 받는 종이 객체는 `confirm_power`가 확정되는 순간 `paperSpawn`에서 생성된다.
- 따라서 재시도 중 월드가 계속 움직여도, 플레이어가 조준하는 동안 종이가 이동 장애물에 맞아 억울하게 실패하는 상황은 발생하지 않아야 한다.

### 7.3 액션 맵

- `aim_x`: 좌우 조준 이동
- `aim_y`: 상하 조준 이동
- `start_power`: 파워 게이지 시작
- `confirm_power`: 파워 게이지 확정
- `retry`: 스테이지 재시도
- `pause`: 일시정지

PC에서는 `start_power`와 `confirm_power`를 같은 입력으로 재사용할 수 있다. 모바일에서는 같은 버튼을 두 번 누르는 방식으로 처리한다.

### 7.4 조작 목표

- PC와 모바일에서 같은 룰을 유지하되, 입력 수단만 다르게 처리한다.
- 조작은 어렵지 않지만 숙련 차이는 존재해야 한다.
- 고의적 난해함이 아니라 계산 가능한 오차 범위를 제공해야 한다.
- 조준 실수와 파워 실수를 분리해 플레이어가 실패 원인을 이해할 수 있어야 한다.

## 8. HUD와 정보 가독성

### 8.1 HUD 배치 권장안

현 시점 기준 권장 HUD는 다음과 같다.

- 상단: 스테이지 번호, 목표 조건, 남은 투척 수, 현재 성공 횟수
- 우측 상단: 바람 HUD
- 하단: 각도 표시 + 파워 게이지

### 8.2 바람 HUD

바람 HUD는 항상 `현재 스테이지의 단일 선풍기` 정보를 표시한다. 복수 풍향 해석이 없으므로 HUD는 단 하나의 기준 값만 보여준다.

- 방향 화살표
- 세기 수치
- 강도 라벨 (`약함`, `보통`, `강함` 등)

HUD는 전역 정보를 제공하고, 플레이필드 내부는 공간 정보를 제공한다.

- HUD 역할: 현재 선풍기의 방향과 세기를 빠르게 확인
- 플레이필드 역할: 그 선풍기의 영향 범위가 어디까지 닿는지 시각적으로 이해

### 8.3 플레이필드 시각 정보

- 선풍기 위치
- 선풍기 영향 범위 파티클 또는 스트릭 효과
- 장애물의 현재 상태
- 쓰레기통 위치와 입구 가시성
- 필요 시 초반 스테이지만 제한된 보조선 제공

### 8.4 정보 표현 원칙

- 실패 원인은 반드시 화면에서 읽혀야 한다.
- 텍스트보다 공간 배치와 간결한 아이콘을 우선한다.
- 모바일에서는 HUD가 시야를 가리지 않아야 한다.

## 9. 물리와 판정 톤

### 9.1 물리 톤

물리 톤은 `하이브리드형 아케이드`로 설계한다.

- 기본은 읽기 쉬운 아케이드 성격
- 후반으로 갈수록 바람과 장애물의 영향이 더 크게 느껴짐
- 현실처럼 보이되, 밸런싱 가능한 단순 모델을 유지
- 충돌은 자연스럽게 반응하지만, 실제 종이보다 약간 더 읽기 쉬운 탄성/감쇠를 사용한다.

### 9.2 권장 기본 변수값

아래 값은 첫 구현에서 권장하는 기준값이다. 단위는 `가상 단위(unit)` 기준이며, `1 unit`은 화면상 약 `1m`에 대응하는 내부 단위로 본다.

| 항목 | 권장값 | 설명 |
| --- | ---: | --- |
| 고정 스텝 `dt` | `1/60 s` | 결정적 시뮬레이션 기준 |
| 최소 발사 속도 | `9.5 unit/s` | 아주 약한 샷의 시작값 |
| 최대 발사 속도 | `15.5 unit/s` | 강한 샷의 시작값 |
| 중력 `gravityY` | `-22.0 unit/s^2` | 실제보다 강해 읽기 쉬운 포물선 생성 |
| 선형 드래그 `linearDrag` | `1.35 1/s` | 공기저항 느낌과 과속 억제 |
| 최대 낙하 속도 | `-18.0 unit/s` | 급가속 낙하 방지 |
| 장애물 반발계수 | `0.22` | 장애물 충돌 시 약하게 튕김 |
| 림 반발계수 | `0.34` | 쓰레기통 입구에서 조금 더 살아 있는 반응 |
| 접선 감쇠 | `0.84` | 충돌 후 옆방향 속도 일부 유지 |
| 최소 분리 속도 | `2.4 unit/s` | 장애물에 붙거나 멈추는 현상 방지 |
| 최대 비행 시간 | `4.0 s` | 장시간 비행 방지 |
| 성공 래치 시간 | `120 ms` | 쓰레기통 내부 성공 확정 시간 |
| 입구 보정 반경 | `0.18 unit` | 과한 흡입 없이 자연스러운 허용 오차 |

### 9.3 바람 세기별 권장값

| 강도 | 목표 횡속도 | 반응 속도 | 중력 배율 |
| --- | ---: | ---: | ---: |
| 약함 | `1.2 unit/s` | `6.0 1/s` | `0.96` |
| 보통 | `2.2 unit/s` | `7.0 1/s` | `0.92` |
| 강함 | `3.4 unit/s` | `8.0 1/s` | `0.88` |

설명:
- 바람은 상수 가속보다 `목표 횡속도에 수렴`하는 방식으로 적용한다.
- 이 방식은 뭉친 종이가 바람을 타며 자연스럽게 옆으로 밀리는 느낌을 만들기 쉽다.
- 동시에 튜닝과 예측 가능성이 높아 완전 예측형 규칙과 잘 맞는다.
- 위 표의 `목표 횡속도`는 절대값 기준이다. 실제 스테이지 데이터의 `targetLateralSpeed`는 좌우 방향까지 포함한 부호값으로 넣는다.

### 9.4 판정 톤

판정은 `중간 판정`으로 한다.

- 각도와 파워를 맞춰야 하지만 약간의 오차는 허용
- 통 입구 근처 성공은 과도하게 빡빡하지 않게 처리
- 완전 자동 흡입처럼 보이지는 않도록 조심
- 충돌 반응은 자연스럽게 보이되, 플레이를 망가뜨릴 정도로 예측 불가능해지지 않게 제한한다.

### 9.5 실패 해석 가능성

플레이어는 다음 중 무엇 때문에 실패했는지 즉시 알 수 있어야 한다.

- 바람에 밀렸다.
- 파워가 과했다.
- 파워가 부족했다.
- 장애물에 막혔다.
- 입구 각도가 맞지 않았다.
- 바닥에 닿았다.

실패 원인은 한 번의 투척에 대해 하나만 표시한다. 권장 우선순위는 아래와 같다.

1. `rim_reject`
- `RimContact` 또는 `EntryCandidate`에 진입한 뒤 최종적으로 성공하지 못한 경우

2. `obstacle_block`
- 최근 `250 ms` 안에 장애물 충돌이 있었고, 그 이후 `RimContact`에 진입하지 못한 경우

3. `wind_push`
- 최근 충돌 없이 바람 영향 범위를 통과했고, 최종 실패 위치의 좌우 편차가 목표 corridor 밖으로 크게 벗어난 경우

4. `power_low` / `power_high`
- 최근 충돌 없이 바닥 또는 유효 범위 이탈로 종료되었고, 종료 시 `z`가 쓰레기통 평면보다 유의미하게 짧거나 긴 경우

5. `ground_hit`
- 위 조건 어디에도 해당하지 않는 기본 실패

이 우선순위는 플레이어에게 보여주는 사유와 로그/텔레메트리 사유에 동일하게 적용한다.

2. 강한 단일 바람

- 바람 자체는 항상 단일 기준으로 유지하고, 난도는 장애물과 배치에서 올린다.

## 12. 시스템 아키텍처

### 12.1 권장 기술 스택

- Phaser
- TypeScript
- Vite
- DOM 오버레이 기반 HUD
- 커스텀 고정 스텝 시뮬레이션

### 12.2 스택 선택 이유

이 게임은 자유로운 3D 이동이 핵심이 아니라, 정면 투척 느낌을 가진 퍼즐 아케이드 구조가 핵심이다. 따라서 진짜 3D 엔진보다, 빠르고 안정적인 2D/2.5D 연출과 입력/시뮬레이션 제어가 유리하다. Phaser는 렌더링과 입력 처리에 적합하고, 실제 판정은 별도의 커스텀 시뮬레이션으로 분리하면 공정성과 튜닝 안정성을 확보할 수 있다.

### 12.4 아키텍처 원칙

- 시뮬레이션과 렌더링을 분리한다.
- 저장 가능한 상태는 렌더러 객체가 아니라 시뮬레이션 데이터여야 한다.
- HUD는 DOM 레이어로 유지해 텍스트 가독성을 확보한다.
- 스테이지 규칙은 데이터 중심으로 관리한다.
- Phaser의 기본 물리 엔진을 소스 오브 트루스로 사용하지 않는다.
- 실제 판정은 커스텀 시뮬레이션이 담당하고, Phaser는 표현 계층에 집중한다.
- 재시도는 월드를 리셋하지 않고 투척 상태만 리셋한다.

## 13. 시뮬레이션 모델

이 게임은 진짜 3D 탄도 시뮬레이션보다 `정면 투척 느낌을 주는 단순화 모델`이 중요하다.

### 13.1 좌표계와 단위

- `x`: 좌우 편차
- `y`: 높이
- `z`: 전진 거리
- `1 unit`은 내부적으로 약 `1m`에 해당하는 가상 단위로 취급한다.
- 바닥 기준면은 `floorY = 0`으로 둔다.

### 13.2 투영 방식

- 시뮬레이션은 3축으로 계산한다.
- 렌더링은 `준 FPS형으로 보이는 2.5D 투영`으로 표현한다.
- 즉, 진짜 3D 카메라가 아니라 가짜 원근 투영을 사용한다.
- 화면상 쓰레기통 입구의 보이는 폭과 실제 충돌 폭은 완전히 같게 두지 않고, 판정 폭이 약간 더 작게 유지되도록 한다.
- 권장 비율: `시각 입구 폭 = 실제 판정 폭의 1.08배`

### 13.3 발사 벡터 계산

권장 공식은 아래와 같다.

```ts
const power01 = clamp((power - minPower) / (maxPower - minPower), 0, 1);
const launchSpeed = lerp(9.5, 15.5, power01);

const yaw = degToRad(yawDeg);
const pitch = degToRad(pitchDeg);

const dir = normalize({
  x: Math.sin(yaw) * Math.cos(pitch),
  y: Math.sin(pitch),
  z: Math.cos(yaw) * Math.cos(pitch),
});

velocity = dir * launchSpeed;
position = paperSpawn;
```

권장 조준 범위:
- `yaw`: `-18deg` ~ `18deg`
- `pitch`: `18deg` ~ `54deg`

이 범위는 아케이드 느낌을 유지하면서도 지나치게 평평한 직사나 과도한 고각 샷을 방지한다.

### 13.3.1 종이 충돌 프록시와 스폰

- 실제 시뮬레이션에서 종이는 회전 없는 `단일 구형 프록시`로 취급한다.
- 시각 표현용 종이 스프라이트의 회전, 찌그러짐, 흔들림은 판정과 분리한다.
- 모든 충돌, 림 스침, 입구 통과, 바닥 판정은 `종이 중심 + 반지름` 기준으로 계산한다.
- 권장 기본값:
  - `paperRadius = 0.11 unit`
  - `paperSpawn = { x: 0.0, y: 1.45, z: 0.65 }`
- `paperSpawn`는 모든 스테이지에서 동일한 기본값을 쓰되, 카메라 연출상 필요하면 스테이지 데이터로 오버라이드할 수 있다.

### 13.4 고정 스텝 업데이트

매 프레임이 아니라 고정 스텝으로 갱신한다.

```ts
const dt = 1 / 60;

velocity.y += gravityY * gravityScale * dt;
velocity *= max(0, 1 - linearDrag * dt);
velocity.x += (windTargetX - velocity.x) * windResponse * dt;
velocity.y = max(velocity.y, maxFallSpeed);
position += velocity * dt;
```

설명:
- `gravityScale`은 바람 범위 안에서만 약하게 줄어든다.
- `windTargetX`는 선풍기 방향과 세기에 의해 결정된다.
- 상수 가속이 아니라 목표 횡속도 수렴 방식을 사용해 종이가 바람을 타는 느낌을 만든다.

### 13.5 바람 적용 방식

- 바람은 단일 선풍기 기준으로만 존재한다.
- 종이가 선풍기 영향 범위에 진입하면 `x` 편차와 낙하 타이밍에 영향을 준다.
- 영향 범위 밖에서는 바람이 적용되지 않는다.
- 바람 범위 안에서는 `gravityScale`이 약간 감소해 종이가 조금 더 떠 있는 듯한 감각을 만든다.

구현 해석을 고정하기 위해 아래를 명시한다.

- `directionDeg`는 월드 `XZ` 평면 기준의 수평 yaw다.
- `0 deg`는 화면 정면인 `+z`, `+90 deg`는 플레이어 기준 오른쪽인 `+x`, `-90 deg`는 왼쪽인 `-x`를 의미한다.
- MVP에서 실제 비행에 적용하는 바람 속도는 `targetLateralSpeed` 하나뿐이며, 이는 부호를 가진 `x`축 목표 횡속도다.
- 즉, 첫 버전에서는 바람이 `z`축 전진 속도를 직접 밀거나 당기지 않는다. 정면 투척의 읽기 쉬움을 위해 `x` 편차와 `gravityScale`만 조절한다.
- `directionDeg`는 선풍기 시각 연출과 영향 범위의 방향 정렬에 사용한다.
- `influenceShape`는 확장용 필드지만, 첫 버전의 실제 스테이지 데이터에서는 `box`만 사용한다.
- `box` 범위는 `fan.position`을 시작점으로 하여 팬의 local `+z` 방향으로 `influenceLength`만큼 뻗고, local `x` 방향으로 `influenceWidth / 2`, `y` 방향으로 `influenceHeight / 2`를 갖는 직육면체로 정의한다.
- `feather`는 박스 경계의 감쇠 폭이며, 범위 내부에서 경계로 갈수록 `windTargetX`와 `gravityScale` 보정을 선형으로 약화한다.

### 13.6 충돌 응답

장애물 충돌은 자연스럽게 보이되, 플레이어가 보기 어려운 비현실적 정지 상태는 허용하지 않는다.

권장 규칙:
- 장애물 충돌 시 법선 방향 속도는 반사하고, 반발계수 `0.22`를 적용한다.
- 접선 방향 속도는 `0.84` 배로 감쇠한다.
- 쓰레기통 림 충돌은 반발계수 `0.34`를 적용해 조금 더 살아 있는 반응을 준다.
- 충돌 후 속도가 너무 낮아 표면에 붙으려 하면, 표면 바깥 방향으로 최소 분리 속도 `2.4 unit/s`를 강제해 멈춤/끼임을 방지한다.
- 장애물 위에 정지 상태를 허용하지 않는다. 충돌은 항상 `임펄스 기반 일회성 반응`이며, 그 다음 프레임부터 다시 자유 비행 상태로 돌아간다.
- 이동 장애물과 충돌할 때는 `paperVelocity - obstacleSurfaceVelocity`로 상대 속도를 만든 뒤, 그 상대 속도에 대해 같은 반사/감쇠 규칙을 적용하고 마지막에 `obstacleSurfaceVelocity`를 다시 더한다.
- 첫 버전의 이동 장애물은 `x` 또는 `y` 축으로만 움직이며, `z`축 이동 장애물은 지원하지 않는다.

즉, 장애물에 맞으면 `약하게 튕기고 에너지를 잃으며 계속 날아가거나 떨어진다`가 기본이다. `맞는 순간 즉시 아래로 추락` 같은 인위적 반응은 사용하지 않는다.

### 13.7 쓰레기통 진입 상태 머신

쓰레기통 판정은 아래 상태 머신으로 처리한다.

1. `Flying`
- 일반 비행 상태

2. `RimContact`
- 림과 접촉한 상태
- 림 반발계수를 적용해 약하게 튕기거나, 조건이 맞으면 입구로 유도된다.

3. `EntryCandidate`
- 종이 중심이 입구 평면을 위에서 아래로 통과했고,
- 수평 오프셋이 `openingWidth / 2` 안이며,
- 속도가 `entrySpeedMin` 이상 `entrySpeedMax` 이하일 때 진입 후보가 된다.

4. `InsideBin`
- 종이 중심이 내부 성공 볼륨 안에 들어온 상태
- 이 상태가 `120 ms` 이상 유지되거나,
- 내부 깊이 기준(`depthTolerance`) 이상 아래로 내려가면 성공 확정 가능
- 이 상태에 들어간 순간부터 월드 `floorY` 실패 판정은 비활성화된다.
- 대신 쓰레기통 내부 바닥과 내부 벽 충돌은 `bin` 로컬 충돌로 처리한다.

5. `SuccessLatched`
- 성공 확정 상태
- 이후 물리 계산은 계속해도 판정은 유지

권장 기본값:
- `entryAssistRadius = 0.18 unit`
- `entrySpeedMin = 2.0 unit/s`
- `entrySpeedMax = 11.5 unit/s`
- `settleTimeMs = 120`
- `depthTolerance = 0.35 unit`

입구 보정은 순간이동처럼 보이면 안 된다. `entryAssistRadius`는 `EntryCandidate` 진입 폭을 넓히는 용도가 아니라, 이미 `RimContact` 또는 `EntryCandidate`에 들어온 샷에 대해 매우 약한 내부 방향 보정을 줄 수 있는 상한값으로 사용한다. 즉, `플레이어가 보기에 들어간 샷만 들어가게` 해야 한다.

### 13.8 투척 종료 조건

투척은 아래 조건 중 하나를 만족하면 종료한다.

- `SuccessLatched` 상태가 되면 성공 종료
- `y <= floorY`가 되는 순간 즉시 실패 종료
- 화면/시뮬레이션 유효 범위를 벗어나면 실패 종료
- 최대 비행 시간 `4.0 s`를 넘기면 실패 종료

중요한 규칙:
- 바닥에 닿는 순간 실패다.
- 장애물 위에 떨어져 멈추는 상태는 허용하지 않는다.
- 다만 `InsideBin`에 진입한 이후에는 월드 바닥 실패를 적용하지 않는다.
- `SuccessLatched` 이전이라도 쓰레기통 내부 바닥 접촉은 실패가 아니라 내부 정착 과정으로 취급한다.

## 14. 스테이지 데이터 구조

스테이지는 코드 하드코딩이 아니라 데이터로 정의한다. 바람은 단일 선풍기 구조로 고정하고, 난이도 차이는 주로 선풍기 세기와 장애물 배치에서 만든다.

권장 스키마는 아래와 같다.

```ts
type ThemeId = 'room' | 'office' | 'classroom' | 'cafe';

type ObstacleKind =
  | 'static_block'
  | 'moving_block'
  | 'narrow_gate';

interface Vec3 {
  x: number;
  y: number;
  z: number;
}

interface PaperConfig {
  spawn: Vec3;
  radius: number;
}

interface AimConfig {
  yawMinDeg: number;
  yawMaxDeg: number;
  pitchMinDeg: number;
  pitchMaxDeg: number;
  defaultYawDeg: number;
  defaultPitchDeg: number;
}

interface PowerConfig {
  mode: 'ping_pong';
  minPower: number;
  maxPower: number;
  gaugeSpeed: number;
  startPower: number;
}

interface PhysicsConfig {
  fixedDtSec: number;
  gravityY: number;
  linearDrag: number;
  maxFallSpeed: number;
  obstacleRestitution: number;
  rimRestitution: number;
  tangentialDamping: number;
  minSeparationSpeed: number;
  maxFlightTimeMs: number;
}

interface FanConfig {
  enabled: boolean;
  position: Vec3;
  directionDeg: number;
  strength: number;
  strengthLabel: 'weak' | 'medium' | 'strong';
  targetLateralSpeed: number;
  windResponse: number;
  gravityScaleInZone: number;
  influenceShape: 'box' | 'cone';
  influenceLength: number;
  influenceWidth: number;
  influenceHeight: number;
  feather: number;
  showParticles: boolean;
}

interface ObstacleMotionConfig {
  type: 'none' | 'ping_pong_x' | 'ping_pong_y';
  amplitude: number;
  durationMs: number;
  phaseMs: number;
  timeSource: 'world_time';
}

interface ObstacleConfig {
  id: string;
  kind: ObstacleKind;
  position: Vec3;
  size: Vec3;
  motion: ObstacleMotionConfig;
}

interface BinConfig {
  position: Vec3;
  openingWidth: number;
  openingHeight: number;
  innerDepth: number;
  depthTolerance: number;
  rimThickness: number;
  entryAssistRadius: number;
  entrySpeedMin: number;
  entrySpeedMax: number;
  settleTimeMs: number;
}

interface ClearConditionConfig {
  throwLimit: number;
  requiredSuccesses: number;
}

interface ScoreConfig {
  mode: 'binary_success';
  successValue: 1;
  failureValue: 0;
}

  resetThrowOnly: true;
  keepWorldTimeOnRetry: true;
}

interface AssistConfig {
  showGuideArc: boolean;
  showFailureReason: boolean;
  showAimReticle: boolean;
}

interface StageConfig {
  id: string;
  order: number;
  theme: ThemeId;
  clear: ClearConditionConfig;
  score: ScoreConfig;
  retryPolicy: RetryPolicyConfig;
  aim: AimConfig;
  power: PowerConfig;
  paper: PaperConfig;
  physics: PhysicsConfig;
  fan: FanConfig;
  obstacles: ObstacleConfig[];
  bin: BinConfig;
  assists: AssistConfig;
  cameraPreset: 'semi-fps-default';
  tutorialText?: string;
  clearConditionText: string;
}
```

이 스키마의 핵심은 다음과 같다.

- `fan`은 배열이 아니라 단일 객체다.
- 장애물은 정적, 이동, 좁은 입구의 세 종류만 우선 지원한다.
- 조준 범위와 파워 게이지도 스테이지 데이터로 조절 가능하다.
- 종이의 실제 판정 프록시는 `paper` 데이터로 고정한다.
- `physics`, `score`, `retryPolicy`를 데이터로 분리해 판정 규칙을 명시한다.
- 이동 장애물은 `world_time` 기준으로 움직여 재시도 시에도 끊기지 않는다.

## 15. 점수, 클리어, 재시도 연속성

### 15.1 점수 판정

점수는 단순하다.

- 쓰레기통에 자연스럽게 들어가 성공 확정되면 `1점`
- 그렇지 않으면 `0점`

첫 버전에서는 아래를 사용하지 않는다.

- 별점
- 보너스 점수
- 남은 투척 수 보너스
- 콤보
- 스타일 점수

### 15.2 클리어 조건

- 각 투척의 성공 여부만 누적한다.
- 누적 성공 횟수가 `clear.requiredSuccesses` 이상이면 즉시 스테이지 클리어다.
- 투척 횟수가 `clear.throwLimit`을 초과하면 스테이지 실패다.

### 15.3 재시도 규칙

재시도 시 초기화되는 것:
- 종이 위치
- 종이 속도
- 조준 상태
- 파워 게이지 상태
- 현재 투척 상태

재시도 시 유지되는 것:
- 월드 시간
- 이동 장애물 위상
- 배경 연출 시간
- 선풍기 동작 상태

즉, 재시도는 월드를 끊지 않는다. 플레이어는 `움직이는 세상 속에서 다시 던진다`는 감각을 유지해야 한다.

재시도 공정성은 `월드를 멈추는 방식`으로 확보하지 않는다. 대신 아래 원칙을 적용한다.

- 조준/파워 단계에서는 실제 물리 종이를 생성하지 않는다.
- 실제 종이 프록시는 `confirm_power`가 확정되는 순간에만 `paper.spawn`에서 생성된다.
- 따라서 재시도 중 이동 장애물과 선풍기는 계속 움직이지만, 플레이어가 다시 조준하는 동안 억울한 피격은 발생하지 않는다.
- 월드 시간과 기믹 위상은 유지되며, 리트라이할수록 타이밍 창이 바뀌는 감각도 그대로 남긴다.

## 16. 에러 처리와 피드백

### 16.1 플레이 피드백

- 성공 시 즉시 시각/청각 피드백 제공
- 실패 시 원인을 읽을 수 있는 연출 제공
- 남은 투척 수가 줄어드는 것을 명확히 표시
- 현재 성공 횟수와 목표 성공 횟수를 명확히 표시
- 목표 달성 시 다음 스테이지 진입을 자연스럽게 연결

### 16.2 시스템 레벨 대응

- 스테이지 데이터 오류 시 안전한 실패 처리 필요
- 잘못된 장애물 배치나 선풍기 값은 로드 단계에서 검증 필요
- 모바일 해상도에서 HUD 겹침 발생 시 레이아웃 축소 규칙 필요
- 쓰레기통 입구 및 내부 볼륨 값이 시각 크기와 지나치게 어긋나지 않게 검증 필요

## 17. 테스트 전략

### 17.1 검증 목표

- 같은 입력이면 항상 같은 결과가 나오는 결정적 시뮬레이션 유지
- 클리어 가능한 스테이지 설계 보장
- 모바일/PC에서 동일 규칙 유지
- HUD 가독성 유지
- 쓰레기통 성공 판정이 플레이어 눈에 보이는 결과와 일치

### 17.2 핵심 테스트 항목

- 동일한 입력에서 발사 벡터와 궤적이 항상 동일한가
- 중력, 드래그, 바람 수렴이 의도한 범위에서 작동하는가
- 선풍기 영향 범위 진입 시 의도한 편차가 적용되는가
- 바람 박스 볼륨의 local 축과 `targetLateralSpeed` 부호가 의도대로 해석되는가
- 장애물 충돌 후 종이가 표면에 붙거나 멈추지 않는가
- 이동 장애물 충돌 시 표면 속도를 반영해 상대 속도 기준으로 반사되는가
- 림을 스치고 들어가는 샷이 자연스럽게 성공 또는 반사되는가
- `InsideBin` 진입 후에는 월드 바닥 실패가 억제되고 내부 정착으로 이어지는가
- 바닥에 닿는 순간 즉시 실패 처리되는가
- 이동 장애물 주기와 클리어 조건이 충돌하지 않는가
- 재시도 시 `worldTime`과 장애물 위상이 유지되는가
- 실패 원인 표시가 정의된 우선순위대로 일관되게 선택되는가
- 각도/파워 입력이 플랫폼별로 같은 논리로 동작하는가
- 모바일에서 HUD와 플레이필드가 겹치지 않는가

## 18. MVP 범위

### 18.1 포함 범위

- 스테이지 순차 진행
- 준 FPS형 정면 투척 화면
- 2단계 확정형 입력 기반의 각도 조절 + 파워 게이지 투척
- 우측 상단 바람 HUD
- 단일 선풍기 기반 바람 영향 범위
- 이진 성공/실패 점수 판정
- 제한 투척 수 / 성공 목표
- 성공/실패 결과 화면
- PC/모바일 반응형 대응
- 6개 스테이지

### 18.2 제외 범위

- 손 애니메이션
- 리플레이 시스템
- 메타 성장 시스템
- 복수 선풍기 시스템
- 실시간 랜덤 풍향 변화
- 별점, 스타일 점수, 보너스 점수
- 상점, 업그레이드, 재화 시스템
- 온라인 요소

## 19. 초기 스테이지 구성

### Stage 1
- 단일 선풍기
- 약한 바람
- 장애물 없음
- 목표: 3번 안에 1회 성공
- 학습 목표: 바람 HUD와 기본 투척 감각 이해

### Stage 2
- 단일 선풍기
- 강한 바람
- 장애물 없음
- 목표: 3번 안에 1회 성공
- 학습 목표: 바람 세기에 따른 보정 이해

### Stage 3
- 단일 선풍기
- 강한 바람
- 고정 장애물 1개
- 목표: 4번 안에 1회 성공
- 학습 목표: 직선 경로가 항상 정답이 아님을 이해

### Stage 4
- 강한 단일 선풍기
- 고정 장애물 2개
- 목표: 4번 안에 1회 성공
- 학습 목표: 우회 경로 해석

### Stage 5
- 중간 세기 단일 선풍기
- 이동 장애물 1개
- 목표: 5번 안에 1회 성공
- 학습 목표: 타이밍 요소 도입

### Stage 6
- 강한 단일 선풍기
- 이동 장애물 1개
- 좁은 입구 1개
- 목표: 5번 안에 1회 성공
- 학습 목표: 바람, 타이밍, 정밀 조준의 종합 테스트

## 20. 출시 판단 기준

아래 기준이 충족되면 첫 버전의 방향성이 맞는 것으로 본다.

- Stage 1~3에서 규칙이 직관적으로 학습된다.
- 실패 원인이 플레이어에게 즉시 읽힌다.
- 모바일과 PC에서 입력 불공정이 크지 않다.
- Stage 4~6에서 계산형 퍼즐 재미가 살아난다.
- 바람 HUD가 과잉 설명 없이도 충분히 유용하다.
- 쓰레기통 판정이 플레이어 눈에 자연스럽게 보인다.

## 21. 후속 계획 방향

권장 진행 순서는 아래와 같다.

1. Stage 1~3 수준의 핵심 루프만 먼저 구현
2. 입력 감각, 바람 수치, 판정 감각 튜닝
3. Stage 4~5로 장애물/타이밍 기믹 확장
4. Stage 6으로 종합 난도 완성
5. 테마 아트 다양화

## 22. 결론

`PaperShoot`의 첫 구현은 `Phaser 기반의 준 FPS형 정면 투척 퍼즐 아케이드`로 정의한다. 핵심은 진짜 3D가 아니라, 단일 선풍기 기준의 읽기 쉬운 바람 정보와 계산 가능한 투척 결과를 중심으로 한 퍼즐 구조다. 첫 버전은 2단계 확정형 입력, 커스텀 고정 스텝 시뮬레이션, 자연스러운 충돌 반응, 쓰레기통 진입 상태 머신, 이진 성공 판정, 6개 스테이지의 점진적 난이도 상승을 기준으로 완성도를 확보하는 것이 가장 현실적이다.
